A — Experiment tracking & reproducibility
Infrastructure configuration versioning, sweeps, and hyperparameter logs.
Completed A1 · Structured hyperparameter sweep via experiment.yaml
High MLflow Flower

The repository features a fully implemented sweep controller (src/sweep.py) running systematic grid sweeps over ewc_lambda, lr, etc., using configs/sweep_grid.yaml, logging each run to MLflow.

  • Variables: ewc_lambda (0.0–0.8), lr, batch_size, fl_rounds.
  • Metrics logged: Logs per-round BWT, F1-per-class, and communication bytes to MLflow.
  • Lineage: Uses MLflow 3.x parent/child run relationships linked via the orchestrator.
  • Visualization: Builds convergence plots dynamically via tools/plot_metrics.py.
Completed A2 · Full experiment lineage graph in MLflow 3.x
High MLflow LoggedModel

The repository automatically computes and logs dataset SHA-256 checksums to MLflow, records lineage metadata via log_dataset.py, and captures full model registry parameters.

  • Hashing: Logs dataset checksums (combined defender-a and defender-b flow hashes) at run start.
  • Lineage tags: Tags each run with parent_run_id and warm-start model version aliases.
  • Summaries: Uses mlflow.log_table() for evaluation history summaries.
  • Auto-promotion: Automates alias promotion (champion/challenger) dynamically in the server.
Planned A3 · Snapshot-based experiment reproducibility with LVM-Thin
High Proxmox LVM-Thin

The infrastructure stack utilizes Proxmox VE with LVM-Thin snapshotting. This plan seeks to automate snapshots before each test sequence to establish a clean state recovery route.

  • Pre-flight: Script qm snapshot <vmid> run-$(mlflow_run_id) in orchestrate.py.
  • Rollback: Automate recovery with qm rollback on experiment run failures.
  • Validation: Compare identical runs on restored snapshots to verify complete determinism.
B — Model registry & lifecycle governance
champion/challenger evaluation, metadata tracking, and model promotions.
Completed B1 · Automated champion/challenger promotion pipeline
High MLflow Registry TorchScript

The aggregator server automatically evaluates models against per-class F1 thresholds, BWT forgetting checks, and communication budgets, assigning 'champion' or 'challenger' aliases dynamically.

  • Gating parameters: Enforces minimum per-class F1, requires non-negative BWT, and validates communication byte consumption.
  • Compilation: Exports TorchScript compiled models on successful promotion.
  • Observability: Dispatches detailed validation breakdowns to Telegram channels for real-time monitoring.
Completed B2 · CL task versioning as first-class model metadata
Medium MLflow Avalanche

Model versions are explicitly tagged with the trained scenario sequence (e.g. cl_task_sequence) and complexity scores, ensuring transparent tracking of task order impact.

  • Tags: Registers task sequence details inside MLflow parameters.
  • Historian: Stores previous task accuracy maps to enable precise BWT derivations.
  • Summary Reporting: Extracts accuracy records at peak convergence round.
Completed B3 · Multi-client model divergence tracking
Medium MLflow Flower

Weights are dynamically checked for L2 drift and variance anomalies using 3-sigma z-score limits during the aggregation step, flagging anomalous clients automatically.

  • Metrics: Logs client_X_weight_drift directly to MLflow tracking.
  • Anomalies: Generates warning alerts if divergence surpasses statistical thresholds.
  • Defense: Identifies bad-acting or outlier nodes before weights influence the global model.
C — Data pipeline & quality gates
Real-time data inspection, drift detection, and replay verification.
Completed C1 · Per-batch data quality gate with distribution shift detection
High NFStream check_dataset.py

A data quality gate exists (tools/check_dataset.py) running label count distribution checks before clients start training, alongside client-side real-time Jensen-Shannon divergence (JSD) shift tracking to reject drifted batches.

  • Groundwork: tools/check_dataset.py verifies label counts and flags missing classes on defender ramdisks.
  • Enforcement: orchestrate.py halts runs (abort action) or flags warnings if the client data violates divergence expectations.
  • Real-time rejection: Client-side fit() checks Jensen-Shannon divergence against baselines, aborting updates and logging telemetry.
Completed C2 · Feature drift monitoring with TensorBoard + MLflow
Medium TensorBoard scikit-learn

Per-class feature stats are analyzed and logged using tools/check_features.py; streaming feature distribution stats directly to MLflow and TensorBoard.

  • Diagnostics: tools/check_features.py extracts mean, std, and Z-scores for flow features per attack category, identifying statistical shifts.
  • Z-Score Limits: Flag skews exceeding 3-sigma thresholds against baseline statistical parameters.
  • Integration: Logged parameters to MLflow runs and visualizes active feature distribution histograms on TensorBoard.
Completed C3 · Benchmark dataset replay reproducibility
High tcpreplay CIC-IDS2017

Orchestration fully replicates standard benchmarks by running sequential simulation stages and tracking cumulative datasets with SHA-256 combined checksums.

  • Integrity: Verifies dataset files match hashes with get_dataset_hash() before execution.
  • Sequencing: Controls attack stages (benign, SSH, slowloris, dns_exfil, botnet) with precise schedules.
  • Provenance: Logs hashes into MLflow artifacts to preserve lineage.
D — Training strategy & continual learning research
EWC parameter sweeps, custom memory algorithms, and warm start runs.
Completed D1 · EWC lambda ablation — forgetting vs. plasticity trade-off
High Avalanche EWC

This is implemented via src/sweep.py, enabling grid search over ewc_lambda parameter sweeps to find the optimal trade-off point.

  • Sweeps: Tests lambda ranges [0.0, 0.1, 0.2, 0.4, 0.8] to trace plasticity curves.
  • Tracking: Pairs runs as child instances under a single parent run in MLflow.
  • Outcome: Helps locate the optimal point protecting previous task knowledge without halting new learning.
Completed D2 · CL strategy comparison: EWC vs. GEM vs. Naive baseline
Medium Avalanche PyTorch

The repository fully supports dynamic continual learning strategy selection (EWC, GEM, and Naive SGD baseline) via the cl_strategy factory pattern.

  • GEM: Gradient Episodic Memory restricts weight updates to prevent task loss using rehearsal memory buffers.
  • Naive: Standard SGD training baseline acts as a control to measure forgetting.
  • Configuration: Dynamically parameterized via configs and CLI options. Results are logged with per-class BWT and F1 metrics in MLflow.
Completed D3 · Warm-start vs. cold-start convergence comparison
High MLflow production-strategy

The server and orchestrator fully support --production-strategy resume (warm-start from the champion alias in Model Registry) or fresh (cold-start from scratch).

  • Download: Queries MLflow model registry for active champion versions and loads weights.
  • Lineage: Logs warm-start tags and registers origin model versions.
  • Evaluation: Compares convergence rates and forgetting patterns between warm and cold starts.
E — Security, privacy & robustness
Poisoning simulations, differential privacy, and robust aggregation.
Completed E1 · Data poisoning attack simulation and detection
Medium Kali MLflow

Simulate malicious participants injecting poisoned traffic inputs (label flips or network patterns) to disrupt the global federated classification model.

  • Simulation: Inject mislabeled packets into a target defender node dataset.
  • Divergence checks: Audit weight updates at the aggregator using cosine similarity.
  • Mitigation: Compare convergence resistance across strategies like Trimmed Mean or Krum.
Completed E2 · Differential privacy noise injection at client level
Hard Opacus Flower

Implement client-side differential privacy (DP) during local backpropagation using Opacus to provide mathematical privacy guarantees for local flow datasets.

  • Opacus: Wrap local model optimizer inside dynamic gradient-clipping privacy engines.
  • Tracking: Log accumulated privacy budget parameters (epsilon, delta) across rounds.
  • Trade-off: Sweep noise multipliers to locate optimal privacy vs. F1 balance boundaries.
Completed E3 · Byzantine-robust aggregation: FedAvg vs. Krum vs. Median
Medium Flower server.py

Replace baseline FedAvg with robust strategies (Krum, Coordinate-wise Median, or Trimmed Mean) to resist outlier update interference.

  • Swapping: Subclass Flower's base aggregation strategy classes.
  • Efficacy: Test resilience against poisoning attempts from plan E1.
  • Trace: Log active strategy names inside MLflow metadata tags.

H — Model architecture & neural network research
Exploring alternative model backbones for ETA flow classifiers.
Planned H1 · Architecture search: MLP vs. 1D-CNN vs. Transformer on ETA features
Medium PyTorch Avalanche

CyberDefenseNet is a 3-layer MLP on 32 flat features. The research paper hints at 1D-CNNs and LSTMs for SPLT sequences. Compare architectures directly — same data, same CL strategy, different model.

  • MLP (current): 32→64→32→5. Fast, but misses sequential characteristics.
  • 1D-CNN: Treat flow packet sequences (SPLT) as 1D temporal signals.
  • Transformer: Self-attention over the ETA feature metrics.
Planned H2 · Fisher Information Matrix inspection across FL rounds
High EWC TensorBoard

EWC computes the Fisher Information Matrix (FIM) after each task to decide which weights to protect. Visualize how the FIM evolves across FL rounds — which model parameters become increasingly "important" and which remain plastic.

  • Telemetry: Stream FIM diagonals to TensorBoard histograms per round.
  • Analysis: Evaluate if feature extraction layers solidify faster than output layers.
Planned H3 · Model compression for edge IDS deployment
Medium PyTorch TorchScript

Apply model optimization, pruning, and quantization steps to reduce storage footprints and acceleration times for embedded IDS environments.

  • Quantization: Apply 8-bit dynamic quantization to TorchScript exports.
  • Pruning: Remove nodes containing weak Fisher values (based on Plan H2 maps).
